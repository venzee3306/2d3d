handleCreateUnits(amount: number, note: string): void
Location: App.tsx
Called by: MobileAdminUnits, UnitsManagement (Admin only)
Purpose: Admin creates new units (mints money into the system).

Input:
  amount: number       // Amount in MMK to create (e.g. 1000000)
  note: string         // Reason/note for creation

Preconditions:
  - currentUser must be logged in (null check)
  - Only Admin should call this (enforced by UI, not handler)

Side Effects:
  1. Adds amount to admin's balance in userBalances
  2. Creates Transaction record:
     {
       id: 'tx{timestamp}',
       userId: currentUser.id,
       type: 'admin_create',
       amount: amount,
       balanceBefore: previousBalance,
       balanceAfter: previousBalance + amount,
       timestamp: ISO string,
       note: note
     }
  3. Prepends transaction to transactions[] array
handleTransferUnits(fromUserId: string, toUserId: string, amount: number, note: string): void
Location: App.tsx
Called by: Internal use (direct transfer between any two users)
Purpose: Transfers units directly from one user to another.

Input:
  fromUserId: string   // Source user ID
  toUserId: string     // Destination user ID
  amount: number       // Amount to transfer
  note: string         // Transfer note

Preconditions:
  - fromUser's balance must be >= amount (silently returns if insufficient)

Side Effects:
  1. Deducts amount from fromUserId's balance
  2. Adds amount to toUserId's balance
  3. Creates TWO Transaction records:
     - fromUser: type='transfer_out', relatedUser=toUser
     - toUser: type='transfer_in', relatedUser=fromUser
  4. Both transactions include balanceBefore/balanceAfter snapshots

Validation:
  - Insufficient balance → silent return (no error thrown)
  - Looks up user names from users[] or players[] arrays
handleTransfer(toUserId: string, amount: number, type: 'deposit' | 'withdraw', note?: string): void
Location: App.tsx
Called by: UnitsManagementV2 → TransferUnitsModal, MobileMasterUnits
Purpose: Transfer with explicit deposit/withdraw direction between current user and a downstream user.

Input:
  toUserId: string          // Target user (downstream agent/master)
  amount: number            // Amount to transfer
  type: 'deposit' | 'withdraw'  // Direction of transfer
  note?: string             // Optional note

Behavior by type:
  'deposit' (give units to downstream):
    - currentUser balance -= amount
    - toUser balance += amount
    - currentUser tx: type='transfer_out'
    - toUser tx: type='transfer_in'

  'withdraw' (take units from downstream):
    - toUser balance -= amount
    - currentUser balance += amount
    - toUser tx: type='transfer_out'
    - currentUser tx: type='transfer_in'

Preconditions:
  - currentUser must exist
  - Both users must exist in users[] array
  - No explicit balance check (can go negative!)

Side Effects:
  - Updates userBalances for both users
  - Creates 2 Transaction records
3. PLAYER DEPOSIT APIs (Player → Agent)
handleApproveDeposit(requestId: string, amount?: number): void
Location: App.tsx
Called by: DepositsView, MobileDeposits
Purpose: Agent approves a player's deposit request (gives units to player).

Input:
  requestId: string    // ID of the LocalDepositRequest to approve
  amount?: number      // Optional override amount (uses request.amount if not provided)

Preconditions:
  - Request must exist in depositRequests[]
  - Agent's balance must be >= deposit amount (silently returns if insufficient)

Side Effects:
  1. Updates request status to 'approved' in depositRequests[]
  2. Deducts depositAmount from agent's balance
  3. Adds depositAmount to player's balance
  4. Creates TWO Transaction records:
     - Agent tx: type='deposit_approve', note='Deposit approved for {playerName}'
     - Player tx: type='transfer_in', note='Deposit approved'

Data Flow:
  depositRequests[] → filter by requestId → update status
  userBalances[agentId] -= amount
  userBalances[playerId] += amount
  transactions[] ← [agentTx, playerTx, ...existing]

4. UNIT DEPOSIT REQUEST APIs (Agent → Master, Master → Admin)
handleRequestUnitDeposit(requestData: Omit<UnitDepositRequest, 'id' | 'requestedAt' | 'status' | 'requesterId' | 'requesterName' | 'requesterRole' | 'approverId' | 'approverName'>): void
Location: App.tsx
Called by: BuyUnitsPaymentFlow (via MobileDashboard, MobileMasterDashboard, MasterDashboard), DashboardView inline modal
Purpose: Creates a new unit deposit request that auto-routes to the correct upstream approver.

Input (what the UI provides):
  {
    amount: number           // Amount requested in MMK
    paymentMethod: string    // Payment method key (e.g. 'kpay', 'wavepay')
    transactionId: string    // Payment proof transaction ID
    paymentScreenshot?: string  // (legacy, not sent from new forms)
    note?: string            // Optional note
  }

Auto-populated fields:
  - id: 'unitdep{timestamp}'
  - requesterId: currentUser.id
  - requesterName: currentUser.name
  - requesterRole: currentUser.role ('agent' or 'master')
  - status: 'pending'
  - requestedAt: ISO timestamp

Routing Logic:
  If currentUser.role === 'agent':
    → approverId = currentUser.parentId (their Master)
    → approverName = Master's name
  If currentUser.role === 'master':
    → approverId = admin user's ID (first user with role 'admin')
    → approverName = Admin's name

Side Effects:
  - Prepends new UnitDepositRequest to unitDepositRequests[]
  - No balance changes at this point (pending request)

Output UnitDepositRequest:
  {
    id: string
    requesterId: string
    requesterName: string
    requesterRole: 'agent' | 'master'
    approverId: string
    approverName: string
    amount: number
    paymentMethod: string
    transactionId: string
    paymentScreenshot?: string
    status: 'pending'
    requestedAt: string
    processedAt?: string
    processedBy?: string
    note?: string
    rejectionReason?: string
  }
handleApproveUnitDeposit(requestId: string): void
Location: App.tsx
Called by: MasterDashboard, MasterDepositsView, MobileMasterDashboard, MobileMasterDeposits, AdminDashboard, WithdrawalsView (Admin), MobileAdminDashboard, MobileBankQRManagement
Purpose: Approver (Master/Admin) approves a unit deposit request.

Input:
  requestId: string    // ID of the UnitDepositRequest to approve

Preconditions:
  - Request must exist in unitDepositRequests[]
  - currentUser must be logged in
  - Approver's balance must be >= request.amount (silently returns if insufficient)

Side Effects:
  1. Updates request in unitDepositRequests[]:
     - status → 'approved'
     - processedAt → ISO timestamp
     - processedBy → currentUser.id
  2. Balance changes:
     - Approver balance -= request.amount
     - Requester balance += request.amount
  3. Creates TWO Transaction records:
     - Approver tx: type='unit_deposit_approve'
       note: 'Unit deposit approved for {requesterName} - {paymentMethod}'
     - Requester tx: type='transfer_in'
       note: 'Unit deposit approved - {paymentMethod}'

Example flow:
  Agent1 requests 100,000 from Master1
  → Master1 approves
  → Master1 balance: 2,500,000 → 2,400,000
  → Agent1 balance: 500,000 → 600,000
  → 2 transactions logged

    - betNumber: item.number
    - amount: item.amount
    - round: 'Morning' (hardcoded)
    - time: current time (HH:MM format)
    - status: 'Pending'

12. TRANSACTION TYPE REFERENCE
Transaction.type values and their meaning:

'admin_create'         — Admin minted new units into the system
'transfer_in'          — Received units from another user
'transfer_out'         — Sent units to another user
'deposit_approve'      — Agent approved a player's deposit request
'deposit_request'      — (Unused in current handlers — reserved)
'unit_deposit_approve' — Master/Admin approved a unit deposit request
'unit_deposit_request' — (Unused in current handlers — reserved)
'withdrawal_approve'   — Master/Admin approved a withdrawal request
'withdrawal_request'   — (Unused in current handlers — reserved)

14. ID GENERATION PATTERNS
User IDs:        'u{Date.now()}'           // e.g. 'u1709312345678'
Player IDs:      'p{Date.now()}'           // e.g. 'p1709312345678'
Transaction IDs: 'tx{Date.now()}-from/to'  // e.g. 'tx1709312345678-from'
                 'txn_{Date.now()}-out/in'  // Alternative pattern used in handleTransfer
Unit Deposit:    'unitdep{Date.now()}'     // e.g. 'unitdep1709312345678'
Withdrawal:      'with{Date.now()}'        // e.g. 'with1709312345678'
Bank Account:    'ba{Date.now()}'          // e.g. 'ba1709312345678'
