Complete Project Blueprint: Myanmar Lottery Player Platform
1. Project Overview
Item	Detail
Platform	Mobile-first Myanmar 2D/3D lottery player app
Stack	React 18 + Tailwind CSS v4 + Vite
Animation	motion (Motion/React) for spring animations
Toasts	sonner for toast notifications
UI Library	Full ShadCN/Radix UI component set under /src/app/components/ui/
Icons	lucide-react
Routing	No React Router — SPA with currentScreen string state in App.tsx
Language	Bilingual: Myanmar (Unicode) / English, toggled via language state
Color Palette	Warm bronze-gold: #6B4F36 → #8B6E4E → #5C3F28 headers, #FAF6F0 background
Auth	Mock login (aungaung / player123)
Persistence	None — all state is in-memory React state (no localStorage yet)

Key Context Functions
Function	Description
login(username, password)	Authenticates against mock player data
logout()	Clears all auth + resets state
updateBalance(amount)	Adds/subtracts from balance
addToCart(items)	Adds items to betting cart
placeBets(sessionId)	Converts cart → bets, deducts balance, creates transaction
placeBetsDirectly(items, sessionId)	Direct bet placement (skips cart), used by betting interface
addTransaction(txn)	Creates a new transaction record
updateTransaction(id, updates)	Updates existing transaction (e.g., pending → approved)
t(key)	Translation lookup (dot-notation keys)
5. Data Models (newMockData.ts)
Player
├── id, name, nameMM, username, password
├── phoneNumber, balance, agentId
├── status ('active' | 'suspended'), createdAt

Bet
├── id, playerId, playerName
├── gameType ('2D' | '3D'), betNumber, amount
├── round ('Morning' | 'Evening'), placedAt
├── status ('Pending' | 'Won' | 'Lost')
├── winAmount?, sessionId?, pattern?

CartItem
├── id, gameMode ('2D' | '3D'), number, amount, pattern?

SavedRecord (betting submission receipt)
├── id, session, timestamp, playerName
├── bets: SavedRecordBetItem[], total

Transaction
├── id, playerId, type ('deposit' | 'bet' | 'win' | 'loss' | 'cashout')
├── amount, balance, description, descriptionMM
├── timestamp, status? ('pending' | 'approved' | 'rejected')
├── relatedBetId?, paymentMethod?

DailyResult
├── id, date, gameType, round, winningNumber, announcedAt

PayoutRate
├── gameType, multiplier (2D=85x, 3D=500x), description

BlockedNumber
├── number, gameType, reason, reasonMM

BankAccount
├── id, playerId, paymentMethod (6 types), accountName
├── accountNumber, bankName?, isDefault, createdAt

AgentPaymentProfile
├── agentId, agentName, agentNameMM
├── paymentMethods: AgentPaymentMethod[] (KBZ/Wave/CB/AYA/One/Bank)
├── lastSynced

UnitDepositRequest
├── id, requesterId, requesterName, requesterRole
├── approverId, approverName, amount, paymentMethod
├── transactionId, paymentScreenshot?, status, requestedAt
├── processedAt?, note?, rejectionReason?
├── type ('deposit' | 'cashout'), accountInfo?
6. Key Feature Systems
A. Betting System (PlayerBettingInterface.tsx)
Full-screen modal with session-aware coloring (AM = amber-gold, PM = bronze-brown)
25 × 2D Quick Add Patterns: group, top, break, back, round, reverse, khwe, khwe-puu, a-puu, sone-puu, ma-puu, pawa, natkha, nyi-ko, ko-nyi, nyi-ko-r, sone-ma, ma-sone, sone-ma-r, sone-sone, ma-ma, sone-kat, sone-kat-r, ma-kat, ma-kat-r
6 × 3D Patterns: a-puu, hundred, series, nyi-ko, ko-nyi, reverse
Pattern categories: A (modal input), B (instant-generate), C (context-dependent on betNumber)
Live bet list with edit/delete per item
Quick action buttons: History, Results, Balance, Settings (navigate & close modal)
Balance deduction via placeBetsDirectly() → creates SavedRecord for history
B. Top Up Flow (DepositQRScreen.tsx) — 3 Phases
Phase 1: Payment method selection grid (6 methods from Agent's profile)
Phase 2: QR code display + form (Unit Amount + Transaction ID only)
Phase 3: Full-screen success animation
C. Request Flows (3 types, all with mock auto-approval)
Flow	Trigger	Creates
Top Up	Dashboard "Buy Units" → DepositQRScreen	UnitDepositRequest + Transaction (pending → approved 3s) + Notification
Buy Units	Transactions "Buy Units" → UnitRequestModal	Same pattern as Top Up
Cash Out	"Cash Out" buttons → CashoutModal	UnitDepositRequest (type: cashout) + Transaction + Notification
All 3 flows:

Create status: 'pending' records immediately
Show yellow "Pending" badge in Transactions tab
Auto-approve after 3 seconds → green "Approved" badge
Update balance on approval
Push real-time notifications
D. Notification System (NotificationPanel.tsx)
Event-driven: notifications created by Top Up, Buy Units, Cash Out, and bet actions
Slide-over panel from right side
Unread counter badge on notification bell
Actions: mark read, mark all read, delete, clear all
Status tracking: pending → approved (matches transaction lifecycle)
E. Results Screen (NewResultsScreen.tsx)
Dark navy theme (contrasts with warm main palette)
2D/3D tabs with historical winning numbers
Dream Book: number-to-meaning lookup
Hot Numbers: frequency analysis of winning numbers
30 days of mock daily results (2D morning + evening, 3D every 5 days)
F. History Screen (NewHistoryScreen.tsx)
Based on SavedRecord objects (complete betting submission receipts)
Shows session, timestamp, player name, individual bets, total
Delete individual records
G. Bank System (BankScreen.tsx + BankAccountModal.tsx)
6 payment methods: KBZ Pay, Wave Pay, CB Pay, AYA Pay, OnePay, Bank Transfer
Full CRUD: Add, Edit, Delete accounts
Set Default account
10 Myanmar banks supported for Bank Transfer type
H. Profile Screen (NewProfileScreen.tsx)
Player info display
Language toggle (EN/MM)
Navigation links to Results, Transactions, Bank
Custom in-app logout confirmation modal

9. Known Gaps / Not Yet Implemented
Gap	Detail
No localStorage persistence	savedRecords, notifications, unitRequests, bankAccounts, and all context state reset on page refresh
No real backend	All flows use mock data and 3-second setTimeout auto-approval
No real QR codes	Canvas-generated deterministic grid patterns (placeholder)
No visual testing done	UI has not been comprehensively tested across devices
UnitRequestModal still used	Transactions screen "Buy Units" still opens the old modal instead of DepositQRScreen
No password change	Profile shows the option but no implementation
No real-time updates	No WebSocket/SSE for live result announcements
No image upload	Receipt/screenshot upload is captured but stored as base64 in memory only
